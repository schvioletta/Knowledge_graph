# R&D Knowledge Graph — горно-металлургическая отрасль

Единая карта знаний, связывающая публикации, эксперименты, материалы,
процессы, условия, оборудование, предприятия, лаборатории/экспертов и
выводы, с гибридным (граф + числовые фильтры + LLM) поиском и интерактивной
визуализацией графа в стиле Connected Papers.

Репозиторий также содержит исходный базовый RAG по PDF (`rag_qdrant.py`),
из которого вырос этот проект.

## Важно: доступ к реальным данным и к внешним LLM API из песочницы

Корпус документов на Яндекс.Диске (см. задание) недоступен из среды
выполнения агента — исходящий трафик на `disk.yandex.ru` и
`cloud-api.yandex.net` блокируется политикой песочницы (HTTP 403 на
уровне egress-прокси, не временная ошибка). Датасет в `data/sample_graph.json`
собран вручную по образцу того, что должен извлекать NLP-пайплайн
из реальных отчётов/статей/патентов, и целенаправленно покрывает все 4
примера запросов из ТЗ (обессоливание воды, циркуляция католита при
электроэкстракции никеля, распределение Au/Ag/МПГ между штейном и шлаком,
закачка шахтных вод), включая намеренный пробел в данных и пару
противоречащих друг другу выводов.

NLP-пайплайн (`backend/nlp_pipeline/`) проверен и вживую прогнан с реальным
LLM (YandexGPT Pro, см. «NLP-пайплайн» ниже) на документах из `data/raw/` —
docx/pptx/pdf, включая OCR для сканов, RU и EN в одном документе. Экстракция
даёт связный граф (сущности реально соединены осмысленными связями
`DESCRIBES_EXPERIMENT`/`USES_MATERIAL`/`USES_PROCESS`/`AUTHORED_BY`/`TAGGED_AS`
и т.д., а не только изолированные узлы Publication). `data/smoke_graph.json` —
результат такого прогона на выборке документов; полный `data/real_graph.json`
получается тем же способом на всём корпусе.

## Архитектура

- **Онтология** (`backend/schema.py`) — сущности `Publication, Experiment,
  Material, Process, Property, Condition, Equipment, Facility, Team, Expert,
  Topic, Conclusion` и связи между ними (`USES_MATERIAL, USES_PROCESS,
  AT_CONDITION, AT_FACILITY, ON_EQUIPMENT, MEASURES_PROPERTY,
  PRODUCES_CONCLUSION, DESCRIBES_EXPERIMENT, CONDUCTED_BY, AUTHORED_BY,
  TAGGED_AS, MEMBER_OF, CONTRADICTS, VALIDATED_BY`). Модель верификации
  фактов — атрибуты `source`, `confidence` (высокая/средняя/низкая), `date`,
  `country` (RU/INTL) на узлах Experiment/Publication/Conclusion. Числовые
  параметры процессов (концентрации, скорости, CAPEX/OPEX и т.д.) хранятся
  прямо в атрибутах узла Experiment — это позволяет фильтровать по
  диапазонам без разрастания графа отдельными узлами на каждое число.
- **Граф-стор — два бэкенда за одним интерфейсом.** Построение графа
  (NLP-пайплайн) и его обслуживание (поиск/визуализация) разделены сознательно:
  - **`backend/graph_store.py` (NetworkX)** — рабочее представление, в которое
    пишет `nlp_pipeline/graph_writer.py`: тысячи мелких мутаций с фаззи-резолвом
    сущностей по имени (`resolve.py` на каждый вызов сканирует все узлы нужного
    типа) — гонять это по сети в внешнюю БД на каждый чанк было бы на порядки
    медленнее, чем in-memory. Загрузка/сохранение в JSON (`data/*.json`).
  - **`backend/graph_store_neo4j.py` (Neo4j, Cypher)** — serving-слой для
    FastAPI: тот же читающий интерфейс (`entities_by_type`, `query_experiments`,
    `experiment_detail`, `gap_matrix`, `contradictions_for`, `vis_subgraph`,
    `neighbors_vis_json` и т.д.), но обход связей — настоящий Cypher
    (`MATCH (e)-[:USES_MATERIAL]->(m) ...`), а не Python-цикл по рёбрам
    NetworkX. Готовый граф заливается в Neo4j одной пакетной операцией
    (`python -m backend.neo4j_sync data/real_graph.json`, см. «Neo4j» ниже) —
    не построчно во время LLM-экстракции.
  - `main.py` и `hybrid_retriever.py` написаны против общего интерфейса (ни
    разу не трогают `.g` — граф NetworkX — напрямую) и одинаково работают с
    обоими бэкендами; переключение — переменной окружения `GRAPH_BACKEND`.
  - Структурные запросы вида «эксперименты с материалом X при процессе Y и
    условии Z» (с OR-альтернативами и числовыми диапазонами), анализ пробелов
    (`gap_matrix`) — какие комбинации материал×условие/процесс/оборудование ещё
    не изучались, поиск противоречий (`contradictions_for`) — реализованы в
    обоих бэкендах с одинаковой семантикой (сверено построчно на demo-датасете,
    см. ниже).
- **Датасет** (`backend/sample_data.py`) → `data/sample_graph.json`.
- **Гибридный ретривер** (`backend/hybrid_retriever.py`):
  - сопоставление сущностей по названию с грубой обработкой русской
    морфологии (падежные окончания) и словарём синонимов RU/EN
    (electrowinning/электроэкстракция, ПВП/печь взвешенной плавки/fluidized
    bed furnace и т.д.);
  - разбор числовых ограничений из вопроса на естественном языке
    («сульфаты ≤300 мг/л», «200–300 мг/л», «не более 1000 мг/дм³»);
  - определение геопризнака (РФ/мировая практика) и временного диапазона
    («за последние 5 лет»);
  - поддержка альтернатив внутри одного запроса («медным/никелевым
    штейном» → ищет оба варианта);
  - структурный запрос к графу + подсветка обнаруженных противоречий в
    выводах;
  - если заданы `YANDEX_API_KEY`/`YANDEX_FOLDER_ID` — финальная стилистическая полировка
    ответа через LLM; без ключа возвращается детерминированный, но
    содержательный ответ с указанием источника и уровня достоверности.
- **NLP-пайплайн** (`backend/nlp_pipeline/`) — сырой документ (docx/pptx/pdf) →
  граф знаний, совместимый с `schema.py`:
  1. `ingest.py` — извлечение текста: PyMuPDF для текстового слоя PDF, OCR-фолбэк
     через `tesseract` (rus+eng) для сканов, `python-docx`/`python-pptx` для
     Word/PowerPoint. Метаданные файла (имя, дата изменения) — заготовка под
     `source`/`date` в модели верификации.
  2. `chunking.py` — язык определяется на уровне абзаца (`langdetect` +
     эвристика по доле кириллицы, если недоступен), а не всего документа —
     важно для русских отчётов с англоязычными аннотациями. Чанки собираются
     по смысловым блокам (абзац/группа/таблица целиком), а не по фиксированному
     числу токенов, чтобы факт «Материал X при Процессе Y дал Z» не рвался
     границей чанка.
  3. `ner_extract.py` — LLM-экстракция: промпт не дублирует онтологию вручную,
     типы сущностей/связей сериализуются прямо из `EntityType`/`RelationType`
     (`schema.py` остаётся источником истины), ответ парсится в pydantic-модели
     — невалидный тип физически не пройдёт парсинг.
  4. `resolve.py` — entity resolution с персистентной таблицей алиасов
     (`data/alias_table.json`): «Электроэкстракция никеля» из одного отчёта и
     «электроэкстракции никеля» из другого не расходятся на два узла (та же
     морфологическая эвристика, что и в поиске — `backend/lexicon.py`).
  5. `validate.py` — SHACL-подобная проверка: допустимость пары
     (source_type, target_type) для каждого типа связи, обязательные поля
     верификации (`source_file`, `confidence`, `date`) на Experiment/
     Publication/Conclusion.
  6. `graph_writer.py` — confidence не «с потолка»: «высокая» = есть явное
     число с единицей измерения И ≥2 независимых источников, «низкая» =
     единичное неявное упоминание, иначе «средняя»; после вставки вывода
     прогоняется `contradictions_for()` по явным `CONTRADICTS`, плюс
     полуавтоматическое сравнение с другими выводами на той же связке
     Материал+Процесс (текстовое сходство без embeddings) — при умеренном
     сходстве помечается `NEEDS_REVIEW`, а не сразу `CONTRADICTS` (ролевой
     модели и ручного review пока нет, поэтому система не берёт на себя
     решение о конфликте, только подсвечивает кандидата).
  7. `sampling.py` + `sections.py` — экономия LLM-вызовов на больших корпусах:
     `sampling.py` отбирает не более N файлов на категорию (категория —
     подпапка внутри `--raw-root`, либо расширение файла как фолбэк),
     стратегией `random`/`largest`/`newest`/`most_tables`, с фиксированным
     `--seed` для воспроизводимости. `sections.py` вместо всего документа
     достаёт только аннотацию/методы/результаты/заключение (по заголовкам
     Word-стиля или эвристике «короткая строка без точки в конце»; если
     формальной структуры нет — берёт начало и конец документа как
     приближение). Это включено по умолчанию (`--full-document` — выключить).
  8. `pipeline.py` — CLI-оркестратор всех этапов.
  9. `eval.py` — считает precision/recall/F1 по сущностям и связям (с fuzzy-
     сопоставлением имён) против ручного golden-набора `data/golden/*.json`
     (формат — см. докстринг `eval.py`); сам набор пока не размечен, каталог
     в репозитории отсутствует — это задел на будущее, а не готовая метрика.

  10. `manifest.py` — манифест обработанных файлов (`data/processed_manifest.json`
      по умолчанию, ключ — sha256 содержимого файла): повторный запуск пайплайна
      на тех же файлах пропускает их без единого LLM-вызова, а изменённый файл
      (другой sha256) обрабатывается заново. `--force` игнорирует манифест.
      Заодно `pub_id`/id сущностей в `graph_writer.py` теперь строятся через
      детерминированный `sha1`, а не встроенный `hash()` — у Python `hash()` строк
      рандомизирован per-process (`PYTHONHASHSEED`), из-за чего id одного и того же
      файла/имени сущности отличались между запусками и плодили дубли узлов при
      повторной обработке вместо апдейта существующих.

  ```bash
  export YANDEX_API_KEY=...
  export YANDEX_FOLDER_ID=...   # YANDEX_MODEL по умолчанию "yandexgpt-5-pro"

  # экономный режим (по умолчанию): только ключевые секции, опционально выборка по категориям
  python -m backend.nlp_pipeline.pipeline data/raw/**/* --per-category 3 --strategy random --out data/real_graph.json

  # повторный прогон на том же корпусе: файлы без изменений пропускаются автоматически
  # (см. data/processed_manifest.json); --force переобрабатывает всё принудительно
  python -m backend.nlp_pipeline.pipeline data/raw/**/* --per-category 3 --strategy random --out data/real_graph.json --force

  # весь корпус целиком, без выборки и без обрезки до секций
  python -m backend.nlp_pipeline.pipeline data/raw/*.docx data/raw/*.pptx data/raw/*.pdf --full-document --out data/real_graph.json

  # смоук-тест: первые 20 файлов — быстро проверить, что пайплайн отрабатывает,
  # прежде чем гонять LLM на всём корпусе
  python -m backend.nlp_pipeline.pipeline data/raw/**/* --limit 20 --out data/smoke_graph.json

  python -m backend.nlp_pipeline.eval data/golden   # precision/recall, если golden-набор размечен (см. eval.py)

  # подставить нужный граф в бэкенд (сначала смоук-граф для проверки, затем боевой real_graph.json)
  GRAPH_DATA_PATH=data/smoke_graph.json uvicorn backend.main:app --reload --port 8000
  GRAPH_DATA_PATH=data/real_graph.json uvicorn backend.main:app --reload --port 8000
  ```

  На 5 реальных файлах в `data/raw/` секционный режим даёт **13 вызовов LLM
  вместо 322** при обработке целиком (замерено этой командой без LLM-ключа —
  подсчёт чанков не требует реального вызова API, см. `--full-document` для
  сравнения).

- **Neo4j (Cypher-хранилище графа)** — `backend/graph_store_neo4j.py` +
  `backend/neo4j_sync.py`. Единая метка `:Entity` на все узлы (тип — свойство
  `type`, не метка) и типы связей, подставленные в текст запроса напрямую —
  осознанное упрощение без плагина APOC (динамические метки/типы связей без
  него не параметризуются, а без APOC контейнер Neo4j поднимается проще и
  быстрее); безопасно, потому что имена связей всегда берутся из фиксированного
  `RelationType` в `schema.py`, а не из пользовательского ввода.

  ```bash
  cp .env.example .env   # либо просто export NEO4J_PASSWORD=...
  docker compose up -d neo4j                      # поднимет Neo4j на bolt://localhost:7687
  python -m backend.neo4j_sync data/sample_graph.json   # или data/real_graph.json — заливка одним пакетом
  GRAPH_BACKEND=neo4j uvicorn backend.main:app --reload --port 8000
  ```

  Проверено вживую (Docker поднят, граф залит, оба бэкенда сравнены построчно
  на `data/sample_graph.json`, 95 узлов/149 рёбер): `/api/health`, `/api/graph`,
  `/api/search` (включая числовые фильтры и обнаружение `CONTRADICTS`),
  `/api/gaps`, `/api/graph/{id}/neighbors`, `/api/timeline` дают идентичный
  результат на NetworkX и на Neo4j — переключение бэкенда не меняет поведение
  API, только то, чем именно выполняется обход связей (Python-цикл vs Cypher).
  Neo4j Browser для ручных Cypher-запросов — http://localhost:7474.

  Пример «настоящего» Cypher-обхода цепочки материал→процесс→оборудование→результат:
  ```cypher
  MATCH (e:Entity {type:'experiment'})-[:USES_MATERIAL]->(m:Entity),
        (e)-[:USES_PROCESS]->(p:Entity),
        (e)-[:ON_EQUIPMENT]->(eq:Entity),
        (e)-[:PRODUCES_CONCLUSION]->(c:Entity)
  WHERE m.name CONTAINS 'штейн'
  RETURN m.name, p.name, eq.name, c.name
  ```

- **FastAPI** (`backend/main.py`): `/api/graph`, `/api/graph/{id}`,
  `/api/graph/{id}/neighbors`, `/api/search`, `/api/gaps`, `/api/timeline`.
  `GRAPH_BACKEND=networkx` (по умолчанию) или `neo4j` — см. выше.
- **Frontend** (`frontend/`, React + Vite + react-force-graph-2d):
  интерактивный force-directed граф в тёмной теме — клик по узлу
  разворачивает соседей, поиск подсвечивает путь рассуждения частицами
  вдоль рёбер, тумблер «Показать пробелы в данных» рисует полупрозрачные
  пунктирные узлы для непокрытых комбинаций, связи `CONTRADICTS`
  выделяются красным пунктиром, связи `NEEDS_REVIEW` (автоматически
  заподозренные конфликты выводов на одной связке материал+процесс, см.
  `graph_writer._flag_similar_conclusions` — это то, что реально
  генерируется на реальных данных, в отличие от явного `CONTRADICTS`,
  который зависит от того, увидит ли LLM оба конфликтующих факта в одном
  чанке) — жёлтым пунктиром, легенда стилей рёбер под графом, ползунок
  «История во времени» анимирует появление экспериментов хронологически.
  Под строкой поиска — панель «База знаний для чата» (`SourcesPanel.jsx`):
  загрузка файла, добавление ссылки, список источников со статусом и числом
  чанков; правая панель результатов получила третью вкладку «По документам»
  (`ResultsPanel.jsx`) с ответом RAG, бейджем достоверности и карточками
  источников. Кнопка GitHub из шапки убрана (не относится к сценарию).

- **RAG-чат по загруженным документам** (`backend/rag/`) — пользователь
  загружает файл (`.pdf/.docx/.txt`) или ссылку на статью через UI (панель
  «База знаний для чата» под строкой поиска), документ чанкуется тем же
  алгоритмом, что и NLP-пайплайн (`nlp_pipeline/chunking.py`), чанки
  кодируются локальной моделью `sentence-transformers`
  (`paraphrase-multilingual-MiniLM-L12-v2`, RU/EN, без внешнего сервера
  векторной БД и без токенов LLM — эмбеддинги считаются локально и бесплатно).
  Вопрос из той же строки поиска параллельно уходит и в граф (`/api/search`),
  и в RAG (`/api/rag/ask`) — это два независимых источника ответа на разных
  вкладках («Текстовый ответ» vs «По документам»), не подменяющих друг друга.
  - **Хранилище — Neo4j (`Neo4jDocumentStore`, `rag/store.py`), не локальные
    файлы.** Каждый загруженный документ — обычный узел `:Entity
    {type: "publication"}` в ТОМ ЖЕ графе, что и остальная база знаний,
    поэтому автоматически появляется в общей визуализации (`/api/graph`) —
    это требует `GRAPH_BACKEND=neo4j` (см. «Neo4j» выше), иначе видимый
    NetworkX-граф просто не знает о том, что лежит в Neo4j. Текст чанков и
    их эмбеддинги хранятся в служебном свойстве `rag_chunks` (JSON-блоб) —
    оно намеренно исключено из ответов `/api/graph`/`/api/graph/{id}`
    (`_HIDDEN_FROM_VIS` в `graph_store_neo4j.py`), иначе десятки КБ
    эмбеддингов на документ раздували бы каждый запрос графа. Пересборка
    демо/боевого графа (`neo4j_sync.py`) не стирает эти узлы — `DETACH
    DELETE` при ресинке явно исключает `rag_chunks IS NOT NULL`.
  - **Дедупликация — по sha256 извлечённого текста.** Второй файл с тем же
    содержимым не проходит через модель эмбеддингов повторно — токены LLM
    тут и так ни при чём (эмбеддинги локальные), но так экономится время
    и не плодятся дублирующие узлы на графе. UI показывает нейтральное
    уведомление «уже есть в базе», а не тишину.
  - Отказ вместо выдумки: если `store.search()` не находит фрагмент выше
    порога релевантности (`min_score=0.35`), в ответ идёт прямой текст
    «не нашлось информации», LLM с пустым контекстом не вызывается вообще
    (`qa.py`).
  - Промпт требует ссылку `[N]` на конкретный фрагмент под каждым
    утверждением и числом, с явным запретом округлять/пересчитывать цифры.
    Без LLM или при ошибке вызова (`llm_client.get_last_error()`) — честный
    детерминированный фолбэк с указанием причины, а не тишина.
  - Confidence — та же логика, что `infer_confidence` в `graph_writer.py`:
    «высокая» только при сходстве ≥0.6 И подтверждении из ≥2 независимых
    документов, «средняя» при сходстве ≥0.45, иначе «низкая»; при полном
    отсутствии совпадений — отдельное значение «нет данных».
  - Ссылка на статью: PDF по прямому URL или HTML (`rag/ingest_link.py`,
    `BeautifulSoup`).
  - Эндпоинты: `POST /api/documents/upload` (multipart), `POST
    /api/documents/link`, `GET /api/documents`, `DELETE
    /api/documents/{id}`, `GET /api/rag/ask?q=...`.
  - Требует поднятого Neo4j независимо от `GRAPH_BACKEND` для основного
    графа (см. «Neo4j» выше) — старт бэкенда падает сразу и понятно
    (`check_rag_store` в `main.py`), если подключиться не удалось, а не на
    первой попытке загрузить файл.
  - Проверено вживую полным циклом: загрузка PDF → узел появляется в
    `/api/graph` (без утечки `rag_chunks` в payload) → повторная загрузка
    того же файла даёт `duplicate: true`, узел не дублируется → реальный
    LLM-ответ с точной цитатой числа из документа → узел переживает
    `neo4j_sync.py` (пересборка демо-графа) → `DELETE` корректно убирает
    узел из графа.

## Соответствие требованиям ТЗ

| Требование | Статус |
|---|---|
| Онтология домена, граф-хранилище с обходом связей (Cypher/Gremlin) | ✅ реализовано: serving-слой — Neo4j (`graph_store_neo4j.py`, реальный Cypher), поднимается через `docker-compose.yml`; сборка графа NLP-пайплайном остаётся на NetworkX (см. «Neo4j» выше про разделение build/serving и почему) |
| Многопараметрические запросы (материал+процесс+условие+география+период) | ✅ реализовано в `hybrid_retriever.py`, работает на обоих граф-бэкендах |
| Числовые диапазоны и ограничения | ✅ реализовано (разбор `≤/≥/диапазон` из текста запроса; на Neo4j — те же ограничения транслируются в `WHERE` по Cypher) |
| Различение RU/зарубежной практики | ✅ реализовано (атрибут `country`, детектор в тексте запроса, фильтр в UI через ответ) |
| Модель верификации (источник/достоверность/дата) | ✅ атрибуты есть и отображаются в ответе и в панели деталей; workflow ручного review — не реализован |
| Версионирование фактов (обновление при новых источниках) | ✅ реализовано: при повторном упоминании сущности с ДРУГИМ значением атрибута новое значение теперь обновляет факт (раньше — молча отбрасывалось через `setdefault`), а расхождение пишется в `node["_history"]` (атрибут/старое/новое/источник/дата) и показывается в панели деталей на графе. Историчность самого графа как снапшотов во времени (кроме уже готового таймлайна по датам публикаций) — не реализована |
| Визуализация графа, подсветка пробелов | ✅ реализовано |
| Подсветка противоречий (`CONTRADICTS`) | ✅ реализовано (граф + текст ответа); плюс `NEEDS_REVIEW` (авто-заподозренные конфликты между источниками, жёлтым на графе) — это основной сигнал на реальных данных, т.к. явный `CONTRADICTS` требует, чтобы LLM увидела оба конфликтующих факта в одном чанке |
| Синтез ответов а-ля «литературный обзор» (консенсус/разногласия) | ⚠ частично: группировка по источникам и перечисление есть, авто-выделение «консенсус vs разногласие» текстом — только через противоречия, не полноценная сводка |
| RAG-чат по загруженным файлам/ссылкам с постраничным цитированием | ✅ реализовано (`backend/rag/`): загрузка pdf/docx/txt и внешних ссылок через UI, хранение в Neo4j (документы видны на общем графе, дедупликация по хэшу содержимого), ответ строго по найденным фрагментам с `[N]`-цитатами (документ+страница/абзац), явный отказ при отсутствии данных вместо выдумки, confidence по числу/качеству подтверждений — см. раздел выше |
| NLP-пайплайн извлечения сущностей из сырых документов (RU/EN) | ✅ реализовано и прогнано вживую с YandexGPT Pro (`backend/nlp_pipeline/`, docx/pptx/pdf+OCR, chunking по языку, entity resolution, валидация, confidence-эвристика); скрипт для P/R/F1 есть (`eval.py`), но сам golden-набор для него пока не размечен |
| Ролевая модель, разграничение доступа, аудит | ❌ не реализовано (нет времени в рамках хакатон-MVP); в схеме легко добавить `visibility`/`role` на уровне узла и middleware в FastAPI |
| Экспорт PDF/Markdown/JSON-LD | ❌ не реализовано; `/api/search` уже отдаёт структурированный JSON, конвертация в Markdown/PDF — расширение эндпоинта |
| Уведомления о новых публикациях | ❌ не реализовано |
| Дашборды для руководителей | ⚠ частично: `/api/gaps` даёт данные для дашборда покрытия по темам, отдельного UI-дашборда нет |
| Сравнительные таблицы технологий | ⚠ частично: сравнение видно через ответ на запрос с альтернативами («медным/никелевым штейном»), отдельного UI-виджета таблицы нет |
| Мультиязычность (RU/EN) | ⚠ частично: словарь синонимов RU/EN для терминов домена есть, полноценного перевода запросов/документов нет |

Честная оценка: это MVP, демонстрирующий архитектуру и ключевую
пользовательскую петлю (вопрос → граф → подсвеченный путь рассуждения →
источники/достоверность → пробелы/противоречия) на репрезентативном
синтетическом датасете, плюс полностью реализованный и юнит-протестированный
(на реальных документах и на мок-LLM) пайплайн извлечения из сырых
docx/pptx/pdf — он не был прогнан на реальном LLM-ключе только из-за
сетевых ограничений песочницы разработки, не из-за незавершённости кода.
Пункты безопасности (RBAC), экспорта и уведомлений — следующий этап, для
которого архитектура (граф-схема, разделение backend/frontend, FastAPI)
уже подготовлена.

## Запуск

### Backend

```bash
pip install -r requirements.txt
python -m backend.sample_data   # один раз генерирует data/sample_graph.json
uvicorn backend.main:app --reload --port 8000
```

### Frontend

```bash
cd frontend
npm install
npm run dev
```

Откройте `http://localhost:5173`. Фронтенд обращается к бэкенду на
`http://localhost:8000` (переопределяется через `VITE_API_BASE`).

### Примеры запросов (соответствуют примерам из ТЗ)

- «Какие методы обессоливания воды подходят при сульфатах, хлоридах, Ca, Mg,
  Na по 200-300 мг/л и сухом остатке не более 1000 мг/дм3?»
- «Какие решения циркуляции католита при электроэкстракции никеля описаны
  в мировой практике, и какая скорость потока оптимальна?» (эксперименты
  на эту тему намеренно противоречат друг другу — система это подсвечивает)
- «Распределение Au, Ag и МПГ между медным/никелевым штейном и шлаком за
  последние 5 лет»
- «Какие способы закачки шахтных вод в глубокие горизонты применялись в
  России и за рубежом?»
- «Кучное выщелачивание никелевой руды в холодном климате» — намеренный
  пробел в данных (буквально пример из ТЗ), система честно сообщает об
  отсутствии данных вместо галлюцинации.
